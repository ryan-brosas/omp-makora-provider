# PRD: Populate production model defaults — maxTokens, vision config, metadata gaps

**Bead:** omp-model-metadata-defaults-m4k | **Type:** task | **Priority:** P1
**Created:** 2026-06-17 | **Estimate:** 40 min

## Problem

WHEN pi sends a chat completion request THEN output token limits are unbounded BECAUSE all 10 Makora models in `models.json` have `maxTokens: 0`. Pi relies on `maxTokens` to cap `max_tokens`/`max_completion_tokens` in API requests — with zero, no cap is applied, and expensive reasoning models (DeepSeek V4 Pro, GLM 5.1, MiniMax M3) can burn thousands of thinking tokens with no guardrail.

Additionally, WHEN a user sends an image to a multimodal model (Kimi K2.6, Kimi K2.7, MiniMax M3) THEN pi doesn't know the image limits BECAUSE the `vision` config (`maxImagesPerRequest`) is missing from `patch.json`. Without explicit limits, pi may default to a conservative value or allow unbounded images, both of which degrade the user experience.

**Who is affected?** All users of all Makora models. Every chat completion is affected by `maxTokens: 0`. Multimodal users are affected by missing vision config.

**Why now?** The existing beads (omp-test-tool-call-parsers-8l6, omp-tool-call-repair-hardening-307) address tool call repair reliability but not model metadata. This bead fills the last major production-readiness gap: proper output caps and capability declarations. Without it, the provider is functional but not production-grade.

## Scope

- Ensure `custom-models.json` Llama 3.3 70B FP8 has consistent metadata with other models

**Data flow:**
```
models.json (API: maxTokens: 0)  →  patch.json (override: maxTokens)  →  merged model list

```

## Approach

1. **maxTokens via patch.json**: Add `maxTokens` to each model's patch entry in `patch.json`. This is the correct layer because `models.json` is auto-generated from the API (which doesn't provide max output tokens) and `patch.json` is the manual override layer.

2. **vision via patch.json**: Add `vision: { maxImagesPerRequest: 5 }` to the three multimodal models' patch entries. Value of 5 is standard for vLLM deployments and matches common defaults.

3. **update-models.js default**: Change the hardcoded `maxTokens: 0` to `maxTokens: 8192` in `transformApiModel`. This is a sensible fallback that gets overridden by `patch.json` for specific models. It's safer than 0.

4. **contextWindow validation**: GLM 5.1's `contextWindow` from the API is 202752 but should be 200000. This is already patched. No other corrections needed based on known specs.

**Alternatives considered:**
- **Set maxTokens in models.json directly** — Rejected. models.json is auto-generated and would be overwritten by `update-models.js`.
- **Set maxTokens in update-models.js per-model** — Rejected. The script doesn't have per-model knowledge; that's patch.json's job.
- **Use 0 as "unlimited" and let pi handle it** — Rejected. Pi uses maxTokens to set `max_tokens`/`max_completion_tokens` in API requests. 0 means no limit, which is dangerous for reasoning models.
- **Set all models to 4096** — Rejected. Too restrictive for reasoning models that need thinking token budget.

## Risks

| Risk | Likelihood | Impact | Mitigation |
|------|-----------|--------|------------|
| maxTokens too restrictive for some use cases | Low | Medium | Values are defaults; pi may allow per-request override. 16K+ for reasoning models provides ample budget. |
| maxTokens too permissive (cost) | Low | Low | Values are caps, not targets. Models only use what they need. |
| vision.maxImagesPerRequest too low | Low | Low | 5 images is generous for coding workflows; can increase later. |
| API changes model capabilities | Low | Medium | patch.json is the manual override layer; can be updated independently. |
| update-models.js default conflicts with patch.json | None | N/A | patch.json always wins (applyPatch runs after model construction). |
