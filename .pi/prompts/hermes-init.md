You are generating a Hermes operator guide for this pi-core project.

## Prerequisites (CHECK FIRST)

Before doing ANYTHING, verify:

1. Hermes is installed: `which hermes`
   - If missing: STOP. Tell the user: "Hermes not found. Install from https://hermes-agent.nousresearch.com"
2. br workspace exists: `br doctor`
   - If missing: STOP. Tell the user: "Run /init first — no br workspace found."

Do NOT proceed without both.

## Phase 1: Gather Project Context

Read these files (use `read` tool):

1. `.pi/memory/project/project.md` — name, goal, success criteria
2. `.pi/memory/project/conventions.md` — how to build
3. `.pi/memory/project/gotchas.md` — known hazards
4. `.pi/memory/project/tech-stack.md` — technologies, verification commands
5. `.pi/memory/project/decisions.md` — architectural choices
6. `.pi/AGENTS.md` — workflow, skills, memory protocol, guardrails

Run these commands:

```bash
br list --json                         # active/open beads
bv --robot-triage 2>&1 | jq '.triage'  # graph state
bv --robot-plan 2>&1 | jq '.plan'      # execution tracks
ls .pi/skills/                         # available skills
cat .pi/settings.json                  # agent configs
```

## Phase 2: Gather Hermes Context

Run these commands:

```bash
hermes profile list                    # available profiles
hermes skills list                     # installed skills
hermes kanban boards list              # kanban state
```

For each profile (excluding `default`), get details:

```bash
hermes profile show <name>             # description, model, skills
```

## Workflow Mapping

| pi-core Phase | Hermes Action | Command |
|---|---|---|
| /brainstorm | Read existing decisions, suggest next work | `bv --robot-triage`, `bv --robot-suggest` |
| /create | Formalize bead + PRD + artifacts | `br create`, write `.beads/artifacts/<id>/` |
| /plan | Decompose into tasks, write plan.md | `bv --robot-plan`, `bv --robot-impact` |
| /ship | Implement via assigned profile | `hermes kanban create --assignee <profile> --workspace worktree` |
| /verify | Run tests, write evidence | [project test command], `.beads/artifacts/<id>/completion-evidence.json` |
| /review | Review via reviewer profile | `hermes kanban create --assignee <reviewer>` |

| Role | Profile | Description | Recommended For |
|---|---|---|---|
[For each non-default profile: name, description from profile show, role based on description]

If only `default` exists: note that user should create specialized profiles for autonomous workflows.

## Autonomy Policy

### May Do (without asking)

- Pick existing actionable beads from `bv --robot-next`
- Read PRD, plan, context-capsule for claimed beads
- Dispatch implementation to assigned profiles via kanban
- Run tests and write completion evidence
- Block and ask Ryan when uncertain

### Ask First

- Create new beads (propose title, type, priority)
- Change architecture or modify `.pi/` config
- Close P0 or P1 beads (needs human confirmation)
- Modify `HERMES.md` or `.pi/AGENTS.md`

### Never

- Skip workflow phases (/brainstorm → /create → /plan → /ship → /verify → /review)
- Close beads without `completion-evidence.json`
- Use `--yolo` by default
- Mutate `~/.hermes/` config from this project
- Treat Hermes Kanban as replacing br (br is source of truth)
- Access files outside project root without explicit instruction

## Evidence Contract

Every completed task must have:

1. `.beads/artifacts/<id>/completion-evidence.json` with:
   - `verification_commands` — commands run + output
   - `requirements_met` — each requirement: PASS/FAIL + evidence
   - `blast_radius_covered` — true/false
   - `files_changed` — list
2. `br close <id> --reason "<specific proof>"`
3. Hermes kanban_complete metadata:
   ```json
   {
     "changed_files": [],
     "tests_run": 0,
     "tests_passed": 0,
     "decisions": []
   }
   ```

## Stop Conditions

| Condition | Action |
|---|---|
| 3 consecutive failures on same task | Block and report to Ryan |
| Tests regress (new failures) | Stop, do not proceed |
| Missing PRD or plan for target bead | Block, ask user to run /create or /plan |
| Architecture decision needed | Block, ask Ryan |
| Unknown profile in kanban create | Check `hermes profile list` first, ask if none fit |

## br/bv Quick Reference

```bash
# State
br list --json                         # all beads
br ready --json                        # actionable
bv --robot-triage                      # triage summary
bv --robot-next                        # top pick

# Lifecycle
br create "title" --type <type> --priority <n> --json
br update <id> --claim --json
br update <id> --notes "COMPLETED: ..." --json
br close <id> --reason "..." --suggest-next --json

# Graph
bv --robot-plan                        # execution tracks
bv --robot-impact <id>                 # blast radius
bv --robot-file-beads <file>           # file history
bv --robot-file-hotspots               # riskiest files
```
```

## Guardrails

- Read live Hermes state — never hardcode profile names
- Read project memory — never invent project context
- Only write to: `HERMES.md`, `.pi/templates/hermes-kanban-card.md`
- If Hermes not installed, report and stop
- If no profiles exist beyond default, warn but proceed
- Re-running overwrites HERMES.md safely (idempotent)
